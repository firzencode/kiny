# 核心引擎设计

> 子项目：`engine/`　|　依据：`docs/reference/kin_spec_draft.md` v0.1.0

## 1. 范围

引擎是一个**纯逻辑、无 UI** 的内核：读取一个 Kiny 项目（一组 `.kin` + `kiny.json`），把它编译并执行为一个可驱动的故事，外加一个终端播放器用于验证。

**做**：parser、静态检查、runtime 状态机、CLI player、覆盖 spec 全部行为的测试。
**不做**：图形界面、音视频实际播放（命令只发事件，不落地）、跨平台打包、编辑器、`kiny.json` 的 `engine` 版本兼容性校验（当前只校验字段存在；引擎当前 `0.0.0`，版本比较无意义，留待后续）。

## 2. 架构

引擎是一条"编译器前端 + 解释器"流水线。核心设计：**读 `.kin`** 与 **跑故事** 彻底分开，中间用 **AST** 与 **事件流接口** 解耦，使 CLI / reader / editor 都接同一接口、永不直接触碰 AST。

```
.kin 文件们                                     运行期
 project   扫描项目根、读 kiny.json、收集 .kin
    │
 parser    每个文件: 文本 → AST（词法 + 语法）
    │
 analyze   合并全局命名空间 + 静态检查 → 已校验程序 | 错误集
    │
    ▼
 runtime   已校验程序 → 有状态的 Story 驱动器
    │         ↑ host 注入内置函数实现 / 命令回调
    ▼
 事件流: {文本} / {选项列表} / {命令}   ←→   输入: 选了第几个选项
```

`project` / `parser` / `analyze` 是**纯函数、无状态**（输入文本，输出 AST 或错误），最易测试。`runtime` 是唯一有状态的部分。

### 2.1 模块划分（`engine/src/`）

| 模块 | 职责 | 依赖 |
|---|---|---|
| `project/` | 递归扫描 `.kin`、解析 `kiny.json`、把文件交给 parser | — |
| `parser/` | 词法 + 语法 → AST；含 AST 类型定义。处理行级控制（`===` `=` `* +` `>` `@if` `~` `~~~` `@命令` `//`）与行内元素（`{}` `<>` `->` `[] ()` `\`） | — |
| `analyze/` | 跨文件合并全局作用域；静态检查（见 §3） | parser |
| `runtime/` | 执行 AST 的状态机：跳转、选项呈现、`>` 分支汇合、粘连、变体计数、JS 求值、作用域、回合数、可复现随机 | analyze |
| `host/` | 内置函数实现（`random`/`seed_random`/`turns`/`turns_since`/变体）与命令出口（`@bg_show` 等）的接口定义 | — |
| `cli/` | 终端播放器 | 以上全部 |

### 2.2 Story 驱动器接口

`runtime` 的产物是一个 `Story` 对象（模型借鉴 inkjs），前端只需四个动作即可驱动：

```ts
story.canContinue          // 还有文本可推进吗
story.continue()           // 推进一步，返回输出事件（文本 / 命令）
story.currentChoices       // 当前要展示的选项（已按条件过滤、含 fallback 逻辑）
story.choose(index)        // 选择，推进到对应分支
```

CLI 的主循环：`canContinue` 就 `continue()` 打印；遇到 `currentChoices` 就让用户选，再 `choose()`。reader 将来同一循环换成 UI。

输出事件类型：
- `text`：一段叙事文本（已完成插值、变体、粘连合并）
- `command`：内置命令调用（名称 + 求值后的实参），交给 host 回调
- 选项不是事件，而是 `currentChoices` 状态

### 2.3 JS 执行

`~` / `~~~` / `{}` 中的 JS 由**宿主直接执行**：把代码片段经 `new Function` 编译为函数，在一个表示当前作用域的对象上调用。

- 全局作用域 + 节点局部作用域（节点含其子节点为一个函数作用域）按 spec §7.2 建模为可嵌套的变量环境
- 内置函数（§12.1）作为该环境里的可见绑定注入
- 变体函数 `seq`/`cycle`/`once`/`shuffle` 按**源码位置**各挂一个计数器，由 runtime 维护（§9）

## 3. 静态检查清单（analyze）

依据 spec 汇总，编译期必须报错的情形：

- 节点名跨全部 `.kin` 重复（§2）；子节点名在同一父节点内重复（§2.1）
- 引用未声明的变量（§7.1 / §7.5）
- 同名变量/函数跨文件重复声明（§7.6）
- 内置函数名被用作变量名 / 参数名 / 选项标签（保留标识符，§7 / §12.1）
- 选项标签重名，或与 `let`/`const` 变量重名（§5.5）
- 带参跳转 `-> 名字(args)` 实参个数与目标节点参数个数不符（§4.2）
- 未知的 `@命令(...)`（§11.2）
- 跳转目标节点 / 子节点不存在
- 节点 / 子节点 / 分支体触底无显式跳转 → **警告**（非错误，§2 / §2.1）

## 4. 构建顺序

自底向上，每层只依赖下层：

```
parser    行级结构 → 块结构 → 行内元素
analyze   跨文件合并 + 全部静态检查
runtime   线性文本+跳转+END → 作用域+JS求值+插值 → 选项
          → 分支体+>汇合 → 粘连 → 变体 → @if → 带参节点+turns+随机
cli       终端播放器
```

## 5. 测试策略

**fixture 黄金测试** 为主。一个 fixture =

```
输入: 一段 .kin（或一个小项目）
脚本: 一串选择序列（每步选第几个选项）
期望: 一份确定性的输出 trace（文本行 / 展示过的选项列表 / 发出的命令）
```

- spec 中的执行示例（§2.1 触底警告、§6 分支汇合、§9 变体推进、§10 粘连缝句、§14 完整示例……）**直接抄为 fixture 的期望输出**，spec 自带验收用例
- trace 用确定性文本序列化格式，金标准存盘，跑测试时比对
- 可复现随机（固定 `seed_random`）让 `shuffle`/`random` 也能进黄金测试
- 静态检查用例：构造违规 `.kin`，断言报出预期错误

判据：能跑通 §14 雾港之夜完整示例，且 spec 各执行示例均有通过测试。

## 6. 边界与未来

- 命令 `@bg_show` 等只发事件、不实际加载资源——落地由 reader 接管
- 沙箱化 JS 执行不在当前范围（宿主直接 `new Function` 执行，见 §2.3）
- `-> DONE` 目前等同 `END`（spec §4.1）
